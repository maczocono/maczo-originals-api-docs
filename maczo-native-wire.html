maczo Originals — Integration wire contract

The exact contract to integrate maczo Originals into your platform. You do two things: call our launch API to open a game for a player, and implement a seamless wallet that we call to move money during play. Everything is JSON, signed with HMAC-SHA256 headers, and uses integer minor units.

Machine-readable schema: the API reference on this site (Swagger UI). Interactive.

1. Roles & directions

  • maczo = game provider. We run the provably-fair games + UI.
  • you (the operator) = the source of truth for the player's balance. Our ledger is a display-only mirror; we never authorise a bet against a balance we computed — your wallet decides.
(1) LAUNCH + CATALOG      you  ──▶  maczo     open a game for a player · sync the game list · end a session
(2) SEAMLESS WALLET       maczo ──▶ you       debit the stake · credit the win · rollback · status · balance

2. Endpoints

You call maczo at your assigned base URL:

Environment Base URL
Sandbox (dev) https://originals-api-dev.maczo.co
Production https://originals-api.maczo.co
POST {base} Purpose
/GameLogin Open a game session for a player; returns a playable URL
/GetGameList Fetch the catalog of games you may offer
/KickPlayer End a player's live session(s)
/ReplayLogin Open a read-only replay of one settled round (support/disputes)

These are JSON, signed with the same header HMAC as the wallet callbacks (§3), and always answer HTTP 200 — classify on the body code (OK / OPERATOR_CONFIG_ERROR / BAD_REQUEST / SIGNATURE_OR_CLOCK). The exact request/response bodies are in the Launch & Catalog API reference on this site. player_ref is your durable player id (charset A-Za-z0-9._$?-, length 1–50) and is echoed verbatim on every wallet callback. game_code is an originals_<id> value from /GetGameList, passed back to /GameLogin.

maczo calls you at the wallet callback base URL you register ({yourWalletBase}{prefix}):

POST Purpose
/debit Place a bet (idempotent on txn_ref)
/credit Settle a win (references the debit via ref_txn_ref)
/rollback Reverse an applied debit
/status Read-only: has a txn_ref applied? (unknown-result recovery)
/balance Display-only balance for a player

The full request/response schema for the wallet you implement is in the API reference on this site.

3. Authentication — HMAC-SHA256 headers

Both directions are signed the same way, with the shared secret you exchange with us (versioned by X-Key-Id, so keys can be rotated with an overlap window):

  • X-Signature — HMAC_SHA256(secret, canonical), lowercase hex
  • X-Timestamp — unix seconds; within ±60 s of now
  • X-Nonce — single-use opaque value (reject a seen nonce)
  • X-Key-Id — selects the signing key

canonical is the newline-joined string (no trailing newline):

METHOD
PATH
X-Timestamp
X-Nonce
sha256hex(rawBody)      # sha256 of the exact request body bytes; empty body -> sha256("")

Reject anything outside the window or with a replayed nonce (fail closed → SIGNATURE_OR_CLOCK).

Rules that catch most integration bugs:

  • PATH is the path only, no query string. On the wallet callbacks you verify, it is the full path including your prefix — e.g. /seamless/debit (not /debit).
  • X-Timestamp is unix seconds, not milliseconds — do not use Date.now() (13 digits fails every time). Keep your clock NTP-synced within 60 s.
  • The secret is used as its raw UTF-8 bytes — not hex/base64-decoded. METHOD is upper-case; the signature is lowercase hex; compare in constant time.
  • Hash the raw bytes as received, before any JSON parsing. Do not parse-then-re-serialize to compute the body hash — a change in key order or whitespace breaks the signature.
  • Money idempotency is always the body txn_ref, never the transport X-Nonce. A retry of the same money move carries the same txn_ref but a new nonce/timestamp/signature. You only need to remember a nonce for ~2× the freshness window (older ones fail the clock check anyway).

4. Money model (non-negotiable)

  • A processed request always answers HTTP 200 — classify the outcome on the body code, never on the HTTP status. (Two transport exceptions on calls to us: rate limiting → HTTP 429 {code:"RATE_LIMITED", retryable:true} with a Retry-After header, and an internal error → HTTP 500 {code:"INTERNAL", retryable:true}. Retry those with backoff. Your wallet, in turn, should always answer 200; we handle your timeouts/5xx as OPERATOR_TIMEOUT/OPERATOR_UNAVAILABLE.)
  • Money is integer minor units + currency + currency_exp (the currency's minor-unit exponent). Never a float. See §4a for the supported currencies and their exponents.
  • currency_exp is present and authoritative on every money callback — always scale to the exact currency_exp we send; never hard-code a currency's decimals.
  • Amounts fit a 64-bit integer. At currency_exp: 8 (crypto) a balance/turnover can exceed JavaScript's 2^53 safe-integer limit — parse and emit every *_minor field as a 64-bit integer / big-integer, never as a float or JS Number.
  • Idempotency key = txn_ref (we derive it; you never mint it — store it unique):
  • same txn_ref and same request → replay: return code: DUPLICATE, duplicate: true, and the live balance in both old_balance_minor and new_balance_minor; move money once.
  • same txn_ref but a different amount/currency/player → code: IDEMPOTENCY_CONFLICT; do not apply (we send request_hash you may use to detect this).
  • A credit (win) or rollback MUST reference the prior applied debit via ref_txn_ref; a settle with no matching debit returns code: TXN_NOT_FOUND (never silent). A losing round never calls your wallet. A debit is settled by at most one credit or one rollback — never both.
  • status(txn_ref) is a read-only query keyed by txn_ref alone (no player_ref). txn_ref is globally unique per operator — key your ledger/dedup by txn_ref, not per-player — so status can resolve it. It returns applied: true|false and moves no money.
  • Unknown / unrecognised code ⇒ fail closed (treat as BAD_REQUEST).

Result codes (you return these on your wallet; classify on code): OK, DUPLICATE, INSUFFICIENT_FUNDS, SESSION_INVALID, SESSION_REPLACED, PLAYER_BLOCKED, ALREADY_SETTLED, IDEMPOTENCY_CONFLICT, TXN_NOT_FOUND, OPERATOR_TIMEOUT, OPERATOR_UNAVAILABLE, RATE_LIMITED, BAD_REQUEST, OPERATOR_CONFIG_ERROR, SIGNATURE_OR_CLOCK.

4a. Currencies, settlement & limits

Settlement currency — the key fact. We call your wallet (debit/credit/rollback/balance) in the exact currency you launch the session with — your enabled currency (e.g. THB, or USDT). There is no FX on the wire: amount_minor is in your currency's minor units at its currency_exp, and every response you return is in that same currency/exponent. You do not receive USD and you do not convert.

Supported currencies + exponents (maczo is authoritative — for a supported currency you may omit currency_exp at launch; we ignore a supplied one and stamp the authoritative value on every callback):

Kind currency codes currency_exp
Fiat (2 dp) USD, EUR, GBP, THB, SGD, MYR, PHP, INR, IDR, BRL, MXN, AUD, CNY, HKD 2
Fiat (0 dp) JPY, KRW, VND 0
Crypto / stablecoin USDT, USDC, DAI, BTC, ETH, BNB, SOL, TRX, TON, XRP, DOGE, LTC 8

currency is the ISO-4217 code for fiat and maczo's symbol for crypto (e.g. USDT, BTC, ETH — not ISO). The 0-dp fiats are a common trap: amount_minor: 500 in JPY is ¥500 (not ¥5). Your enabled currencies are provisioned during onboarding (see the Onboarding guide) — a launch with a currency outside your enabled set is rejected BAD_REQUEST.

Examples across exponents (the launch currency flows unchanged into every callback):

฿1,000.00 THB (exp 2)  ->  { "amount_minor": 100000,        "currency": "THB",  "currency_exp": 2 }
¥1,000    JPY (exp 0)  ->  { "amount_minor": 1000,          "currency": "JPY",  "currency_exp": 0 }
0.00100000 BTC (exp 8) ->  { "amount_minor": 100000,        "currency": "BTC",  "currency_exp": 8 }
12.34567890 USDT       ->  { "amount_minor": 1234567890,    "currency": "USDT", "currency_exp": 8 }

Bet limits. Each game has a min-bet, max-bet, and max-win, enforced per settlement currency (auto-derived, RTP-safe, floored to the currency's exponent) — a bet below the min is rejected BET_TOO_SMALL, above the max BET_TOO_LARGE, and a win above the cap is clipped. You don't configure these per currency; the effective values are visible in the merchant portal.

5. Ordering guarantee

Every bet follows a strict order so money is never at risk: validate the session → debit your wallet and confirm it applied → draw the provably-fair outcome → commit the round → credit any win → reveal the result to the player. A rejected or unfunded bet never draws an outcome, and a win is never shown for an unfunded bet.

Settlement can lag / be retried. A credit or rollback for a debit may arrive seconds-to-minutes after the debit and out of band from live play (a failed settle is parked and retried by reconciliation until you acknowledge it). Honour a credit/rollback against its referenced debit with no expiry, and rely on txn_ref idempotency for the retries. A single debit is resolved by at most one credit or one rollback — never both.

6. Provably fair

Every result is re-derivable from (server_seed, client_seed, nonce) via HMAC-SHA256(server_seed, "client_seed:nonce:cursor"). The server-seed hash is committed before use and revealed on rotation, so any player or you can independently verify any round.

7. Signing keys & the shared secret

  • You generate the secret (e.g. 32 random bytes as hex) and send it to us out-of-band (never on the wire). We register your operator_id and a key_id; you present that key_id as X-Key-Id.
  • The secret is used as its raw UTF-8 bytes in HMAC_SHA256(secret, canonical) — we do not hex- or base64-decode it first. Whatever string we exchange is the key material verbatim.
  • Rotation is two-directional. We can hold more than one active key for you. On calls you sign to us, present the new key_id while the old one still verifies during the overlap. On the wallet callbacks we sign to you, keep accepting both key_ids (map X-Key-Id → the right secret) until we confirm cut-over, then retire the old secret.

8. Callback registration

You register (in the portal, or we set it at onboarding) your wallet callback base URL and a path prefix (default /seamless). We call {base}{prefix}/debit, /credit, etc. The signed PATH in the HMAC canonical is exactly {prefix}/{func} (e.g. /seamless/debit) — the scheme/host are not signed, so your prefix must match on both sides. You also register an inbound IP allowlist for the hosts that call our launch API.

9. Go-live checklist

  1. We register your operator + key_id; you send your secret out-of-band (§7).
  2. Implement the five wallet endpoints (Seamless Wallet API) and verify our signature (§3).
  3. Set your wallet callback base URL + path prefix + inbound IP allowlist (§8).
  4. Test on the sandbox (https://originals-api-dev.maczo.co) end-to-end: /GameLogin → open the returned URL → place a bet → your wallet receives debit then credit/no-call-on-loss → reveal. Confirm your wallet honours idempotency (txn_ref), insufficient-funds, and status.
  5. We approve go-live → your operator flips to active on production.

See Onboarding on this site for the step-by-step.
