maczo Originals — Integration wire contract

The exact contract to integrate maczo Originals into your platform. You do two things: call our launch API to open a game for a player, and implement a seamless wallet that we call to move money during play. Everything is JSON, signed with HMAC-SHA256 headers, and uses integer minor units.

Machine-readable schema: the API reference on this site (Swagger UI). Interactive.

1. Roles & directions

  • maczo = game provider. We run the provably-fair games + UI.
  • you (the operator) = the source of truth for the player's balance. Our ledger is a display-only mirror; we never authorise a bet against a balance we computed — your wallet decides.
(1) LAUNCH + CATALOG      you  ──▶  maczo     open a game for a player · sync the game list · end a session
(2) SEAMLESS WALLET       maczo ──▶ you       debit the stake · credit the win · rollback · status · balance

2. Endpoints

You call maczo at your assigned base URL:

Environment Base URL
Sandbox (dev) https://originals-api-dev.maczo.co
Production https://originals-api.maczo.co
POST {base} Purpose
/GameLogin Open a game session for a player; returns a playable URL
/GetGameList Fetch the catalog of games you may offer
/KickPlayer End a player's live session(s)
/ReplayLogin Open a read-only replay of one settled round (support/disputes)

These are JSON, signed with the same header HMAC as the wallet callbacks (§3), and always answer HTTP 200 — classify on the body code (OK / OPERATOR_CONFIG_ERROR / BAD_REQUEST / SIGNATURE_OR_CLOCK). The exact request/response bodies are in the Launch & Catalog API reference on this site. player_ref is your durable player id (charset A-Za-z0-9._$?-, length 1–50) and is echoed verbatim on every wallet callback. game_code is an originals_<id> value from /GetGameList, passed back to /GameLogin.

maczo calls you at the wallet callback base URL you register ({yourWalletBase}{prefix}):

POST Purpose
/debit Place a bet (idempotent on txn_ref)
/credit Settle a win (references the debit via ref_txn_ref)
/rollback Reverse an applied debit
/status Read-only: has a txn_ref applied? (unknown-result recovery)
/balance Display-only balance for a player

The full request/response schema for the wallet you implement is in the API reference on this site.

3. Authentication — HMAC-SHA256 headers

Both directions are signed the same way, with the shared secret you exchange with us (versioned by X-Key-Id, so keys can be rotated with an overlap window):

  • X-Signature — HMAC_SHA256(secret, canonical), lowercase hex
  • X-Timestamp — unix seconds; within ±60 s of now
  • X-Nonce — single-use opaque value (reject a seen nonce)
  • X-Key-Id — selects the signing key

canonical is the newline-joined string (no trailing newline):

METHOD
PATH
X-Timestamp
X-Nonce
sha256hex(rawBody)      # sha256 of the exact request body bytes; empty body -> sha256("")

Reject anything outside the window or with a replayed nonce (fail closed → SIGNATURE_OR_CLOCK).

4. Money model (non-negotiable)

  • HTTP status is always 200. Classify the outcome on the body code, never on the HTTP status.
  • Amounts are integer minor units (amount_minor) + currency (ISO-4217) + currency_exp (minor-unit exponent). Never a float; full precision (e.g. 8-dp crypto) is supported.
  • Idempotency key = txn_ref. A duplicate returns code: DUPLICATE, duplicate: true, and the live balance — it never re-moves money.
  • A credit (win) or rollback MUST reference a prior applied debit via ref_txn_ref; a settle with no prior debit returns code: TXN_NOT_FOUND (never silent). A losing round is settled internally and does not call your wallet.
  • status(txn_ref) is a read-only query — it must never move money.
  • Unknown / unrecognised code ⇒ fail closed (treat as BAD_REQUEST).

Result codes: OK, DUPLICATE, INSUFFICIENT_FUNDS, SESSION_INVALID, SESSION_REPLACED, PLAYER_BLOCKED, ALREADY_SETTLED, IDEMPOTENCY_CONFLICT, TXN_NOT_FOUND, OPERATOR_TIMEOUT, OPERATOR_UNAVAILABLE, RATE_LIMITED, BAD_REQUEST, OPERATOR_CONFIG_ERROR, SIGNATURE_OR_CLOCK.

5. Ordering guarantee

Every bet follows a strict order so money is never at risk: validate the session → debit your wallet and confirm it applied → draw the provably-fair outcome → commit the round → credit any win → reveal the result to the player. A rejected or unfunded bet never draws an outcome, and a win is never shown for an unfunded bet.

6. Provably fair

Every result is re-derivable from (server_seed, client_seed, nonce) via HMAC-SHA256(server_seed, "client_seed:nonce:cursor"). The server-seed hash is committed before use and revealed on rotation, so any player or you can independently verify any round.

7. Signing keys & the shared secret

  • You generate the secret (e.g. 32 random bytes as hex) and send it to us out-of-band (never on the wire). We register your operator_id and a key_id; you present that key_id as X-Key-Id.
  • The secret is used as its raw UTF-8 bytes in HMAC_SHA256(secret, canonical) — we do not hex- or base64-decode it first. Whatever string we exchange is the key material verbatim.
  • Rotation: we can hold more than one active key for you; sign with the new key_id while the old one still verifies during the overlap, then we retire the old one.

8. Callback registration

You register (in the portal, or we set it at onboarding) your wallet callback base URL and a path prefix (default /seamless). We call {base}{prefix}/debit, /credit, etc. The signed PATH in the HMAC canonical is exactly {prefix}/{func} (e.g. /seamless/debit) — the scheme/host are not signed, so your prefix must match on both sides. You also register an inbound IP allowlist for the hosts that call our launch API.

9. Go-live checklist

  1. We register your operator + key_id; you send your secret out-of-band (§7).
  2. Implement the five wallet endpoints (Seamless Wallet API) and verify our signature (§3).
  3. Set your wallet callback base URL + path prefix + inbound IP allowlist (§8).
  4. Test on the sandbox (https://originals-api-dev.maczo.co) end-to-end: /GameLogin → open the returned URL → place a bet → your wallet receives debit then credit/no-call-on-loss → reveal. Confirm your wallet honours idempotency (txn_ref), insufficient-funds, and status.
  5. We approve go-live → your operator flips to active on production.
